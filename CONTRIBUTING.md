# Contributing to Handle

Thanks for your interest in contributing to Handle.

We welcome bug reports, documentation improvements, test coverage, and well-scoped fixes. We also welcome ideas for improvements. However, please note that **the project roadmap is set by the maintainers**. This means we may decline feature proposals or architectural changes that are not aligned with the current direction of the project.

## Before you start

For anything beyond a small, obvious fix, **please open an issue before writing code**.

Examples of changes that should be discussed first:
- new features
- significant refactors
- public API changes
- dependency changes
- behavior changes
- build, tooling, or CI changes

This helps make sure your effort is aligned with the project and avoids wasted work.

### What happens if I open a PR without prior discussion?

For non-trivial changes, we may close the PR and ask that the proposal be discussed in an issue first.

Small changes such as typo fixes, documentation clarifications, or narrowly scoped bug fixes may not need prior discussion, but when in doubt, open an issue first.

## Good types of contributions

We are especially likely to accept:
- bug reports with clear reproduction steps
- bug fixes for confirmed issues
- documentation improvements
- test coverage improvements
- small UX or developer-experience improvements that fit the current direction

## Contribution process

1. **Open or find an issue**
   - For bugs, please include reproduction steps, expected behavior, and actual behavior.
   - For new ideas, describe the problem, proposed solution, alternatives considered, and why it fits the project.

2. **Wait for maintainer feedback**
   - A maintainer will let you know whether the change is in scope and whether a PR would be welcome.

3. **Fork and create a branch**
   - Use a focused branch for your change.

4. **Make the change**
   - Keep the scope narrow.
   - Avoid unrelated refactors.
   - Update docs and tests where appropriate.

5. **Open a pull request**
   - Link the issue.
   - Explain what changed and why.
   - Note any tradeoffs or follow-up work.

## Pull request guidelines

Please make sure your PR:
- is linked to an issue
- addresses a single concern
- includes tests where appropriate
- updates documentation where appropriate
- does not include unrelated cleanup or refactoring
- keeps commits and diff reasonably focused

PRs that are large, unexpected, or out of roadmap may be declined even if they are technically sound.

## Style and quality

Please follow the conventions already used in the repository:
- code style and formatting
- file organization
- test patterns
- naming conventions
- commit hygiene

If the repository includes linting, formatting, or test commands, please run them before opening a PR.

## Reporting bugs

Please use the bug report issue template and include:
- environment details
- exact reproduction steps
- expected behavior
- actual behavior
- logs or screenshots if helpful

## Security issues

Please **do not** report security vulnerabilities in public issues or pull requests.

Instead, email: support@tonkotsu.ai

See `SECURITY.md` for details.
